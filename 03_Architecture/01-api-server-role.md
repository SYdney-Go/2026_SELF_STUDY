# 범위 1: API 서버는 시스템에서 뭐 하는 애인가

## 한 줄 정의

**API 서버 = 클라이언트(대시보드, 앱, 다른 서버)의 요청을 받아서, 필요한 처리를 하고, 결과를 JSON으로 돌려주는 중간 관문**

관제/VMS 맥락에서 보면:

```text
CCTV/센서/AI 분석
        ↓
   (이벤트 발생)
        ↓
   API 서버  ←── 대시보드/운영자가 "이벤트 목록 줘", "상태 바꿔" 요청
        ↓
   DB (이벤트 저장)
```

API 서버는 **데이터가 저장되는 곳(DB)도, 화면(클라이언트)도 아닙니다.**  
요청을 **받고 → 검증하고 → 처리하고 → 응답하는** 역할입니다.

---

## API 서버가 하는 5가지

| # | 역할 | 관제 이벤트 예시 |
| --- | --- | --- |
| 1 | **요청 받기** | `GET /events` — "이벤트 목록 줘" |
| 2 | **요청 검증** | 필수값 빠졌나, ID 존재하나 |
| 3 | **비즈니스 로직** | 상태를 NEW → IN_PROGRESS로 바꿀 수 있나 |
| 4 | **데이터 접근** | DB에서 이벤트 조회/저장 |
| 5 | **응답 반환** | JSON + HTTP 상태 코드 |

---

## 왜 "API 서버"를 따로 두나

처음엔 "DB에 직접 붙으면 되지 않나?" 싶을 수 있습니다.

```text
[나쁜 그림]
대시보드 ──직접──→ PostgreSQL
```

문제:

- DB 구조가 바뀌면 **모든 클라이언트**를 수정해야 함
- 권한/인증을 클라이언트마다 따로 넣어야 함
- "상태 변경 규칙" 같은 로직이 UI에 흩어짐

```text
[좋은 그림]
대시보드 ──→ API 서버 ──→ PostgreSQL
모바일  ──→     ↑
다른 서비스 ──→  │
              (규칙·인증·검증은 여기서)
```

**API 서버 = 시스템의 "공식 창구"**  
누가 접근하든 같은 규칙, 같은 데이터 형식.

## 관제 이벤트 API — 데이터 흐름

### 이벤트 등록 (POST /events)

```text
[운영자/시스템]          [API 서버]              [DB]
      |                      |                     |
      | POST /events         |                     |
      | { title, severity }  |                     |
      |--------------------->| 1. body 검증        |
      |                      | 2. status=NEW 설정  |
      |                      | 3. INSERT           |
      |                      |-------------------->|
      |                      |<--------------------|
      | 201 + { id, ... }    |                     |
      |<---------------------|                     |
```

### 이벤트 상태 변경 (PATCH /events/1/status)

```text
[대시보드]               [API Server]            [DB]
      |                      |                     |
      | PATCH .../status     |                     |
      | { status: RESOLVED } |                     |
      |--------------------->| 1. id=1 존재?       |
      |                      | 2. 상태 전환 OK?    |
      |                      | 3. UPDATE           |
      |                      |-------------------->|
      | 200 + { id, status } |                     |
      |<---------------------|                     |
```

---



## 이 범위에서 기억할 것

1. API 서버 = **창구**. DB도 UI도 아님.
2. 요청 → 검증 → 로직 → DB → JSON 응답.
3. 관제 시스템에서 "이벤트 조회/상태 변경"은 전부 이 창구를 통함.
4. v1은 작지만, **역할은 실제 관제 시스템과 같음**.
